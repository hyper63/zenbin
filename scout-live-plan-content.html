📊 Overview

Scout Live (formerly hyper-live) is a lightweight deployment platform designed specifically for:

  • Vite-built frontends — React, Svelte, Vue, vanilla JS
  • Optional backends — Bun + Hono + SQLite sandbox
  • ScoutOS integration — Automatic API proxy with secret injection

Project Location

/Users/mastercontrol/.openclaw/workspace/hyper-live

Core Principles

  1. Vite-first — Optimized for Vite builds (React, Svelte, Vue)
  2. ScoutOS-native — Built-in API gateway for ScoutOS services
  3. Isolated — Per-app sandbox with separate database
  4. Simple auth — First to claim subdomain owns it (publish code)
  5. Pure API — No landing page, just REST endpoints

📡 API Specification

POST /api/deploy

Deploy a new app or update an existing one.

Request 
POST /api/deploy
Content-Type: multipart/form-data

Fields:
├── subdomain: string (required)     # e.g., "myapp" → myapp.scout.live
├── artifact: File (required)        # .zip containing frontend/ and optional backend/
├── code: string (optional)           # Required if subdomain already claimed
├── scoutWorkspace: string (optional) # ScoutOS workspace ID for secrets
└── scoutApiKey: string (optional)    # ScoutOS API key (stored securely)
Response 
HTTP/1.1 200 OK
Content-Type: application/json

{
  "subdomain": "myapp",
  "publishCode": "abc123xyz456def",
  "frontend": "https://myapp.scout.live",
  "backend": "https://api.scout.live/myapp",
  "scoutosProxy": "https://api.scout.live/scoutos",
  "vite": true,
  "created": true
}

GET /api/apps

List all deployed apps (metadata only, no publish codes).

Response 
HTTP/1.1 200 OK
Content-Type: application/json

{
  "apps": [
    {
      "subdomain": "myapp",
      "createdAt": "2026-01-15T10:30:00Z",
      "lastDeploy": "2026-01-20T14:22:00Z",
      "hasBackend": true
    }
  ],

}

DELETE /api/apps/:subdomain

Delete an app and its data.

Request 
DELETE /api/apps/myapp
X-Publish-Code: abc123xyz456def

Frontend Serving

Frontend files are served via subdomain routing:

# Production (custom domain)
GET https://myapp.scout.live/          → frontend/index.html
GET https://myapp.scout.live/assets/*  → frontend/assets/*

# Development fallback (path-based)
GET http://localhost:3001/--/myapp/   → frontend/index.html

Backend Proxy

Backend requests are proxied to isolated sandbox processes:

# Production
GET https://api.scout.live/myapp/todos → backend:31000/todos

# Development
GET http://localhost:3001/api/myapp/todos → backend:31000/todos

ScoutOS API Proxy

All ScoutOS APIs are proxied with automatic authentication:

# From app's backend
POST /api/scoutos/v2/agents/{agentId}
Content-Type: application/json
X-ScoutOS-Key: {optional-override}

{
  "message": "Hello from Scout Live app!"
}

# Forwards to:
# https://scoutos.com/api/v2/agents/{agentId}
# With Authorization: Bearer {stored-api-key}

Frontend Rules

  • frontend/index.html is required as the entry point
  • All files served as static assets
  • SPA routing: all non-file routes fallback to index.html
  • Vite builds detected by assets/[name]-[hash].[ext] pattern

Backend Rules

  • backend/index.ts (or .js) is the entry point
  • Must export a fetch handler or Hono app
  • Runs in sandboxed Bun process with isolated env
  • Per-app storage: /data/{subdomain}.db (SQLite)
  • Access ScoutOS via injected env.SCOUTOS_API_KEY

Example Backend

backend/index.ts 
import { Hono } from "hono";
import { cors } from "hono/cors";

const app = new Hono();
app.use("*", cors());

// Health check
app.get("/__health", (c) => c.json({ ok: true }));

// Your API endpoints
app.get("/todos", async (c) => {
  const db = c.env.DB; // SQLite from env
  const todos = db.query("SELECT * FROM todos").all();
  return c.json({ todos });
});

app.post("/todos", async (c) => {
  const { title } = await c.req.json();
  const db = c.env.DB;
  db.run("INSERT INTO todos (title) VALUES (?)", title);
  return c.json({ created: true });
});

export default app;

🔒 Security Model

Isolation Layers

  1. Process Isolation — Each backend runs in separate child process
  2. Filesystem Isolation — Apps can only access /data/{subdomain}
  3. Database Isolation — Per-app SQLite databases
  4. Resource Limits — Memory, CPU, timeout per request (planned)

Secret Handling

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Scout │────→│ Scout Live │────→│ App │ │ Workspace │ │ Deploy API │ │ Sandbox │ └──────────────┘ └──────────────┘ └──────────────┘ │ │ │ │ Secrets at │ Store hashed │ Inject at │ deploy time │ (never logged) │ runtime └────────────────────┴────────────────────┘
  • API keys stored hashed, never logged
  • Injected at runtime into isolated env
  • Apps access via process env, never in user code

Auth Flow

  1. First deploy to subdomain generates publish code
  2. Subsequent deploys require correct publish code
  3. Publish codes stored as SHA-256 hashes
  4. Future: JWT, OAuth integration planned
